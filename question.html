Q 01
CQRS — Architecture
What does CQRS stand for, and what is the core idea behind separating commands from queries?
✦ Answer
CQRS = Command Query Responsibility Segregation.

The core idea: commands (write operations that change state) and queries (read operations that return data) are handled by completely separate models.

• A Command expresses intent to change state — it returns nothing (or just an acknowledgement). Example: CreateOrderCommand.
• A Query returns data and has no side effects. Example: FindOrderByIdQuery.

This separation lets you optimize each side independently — the write side for consistency and business rules, the read side for fast querying (different DB, different schema).

Project Setup
Q 02
Axon — Setup
In the guide's application.yml, why do we set axon.axonserver.enabled: false? What happens if we don't?
✦ Answer
By default, Axon Framework tries to connect to Axon Server (a separate process) on startup for message routing and event storage.

Setting axon.axonserver.enabled: false tells Axon to use its embedded infrastructure instead:
  • An in-process EmbeddedEventStore backed by JPA (H2 in dev)
  • An in-memory SimpleCommandBus and SimpleEventBus
Without this flag and without Axon Server running, the application fails to start with a connection refused error. For local development without external infrastructure, always disable it. 
# application.yml
axon:
  axonserver:
    enabled: false   # use embedded event store (JPA)

Q 03
Axon — Commands
What is @TargetAggregateIdentifier and what goes wrong if you forget it on a command?
public record ShipOrderCommand(
    String orderId,        // ← missing annotation!
    String shippingAddress
) {}
✦ Answer
@TargetAggregateIdentifier marks the field that holds the aggregate instance ID. Axon's CommandBus reads this value to route the command to the correct aggregate instance in the repository.

Without it: Axon cannot determine which aggregate to load. The dispatch throws an AxonConfigurationException at startup — or the command gets routed to a wrong/new aggregate instance, causing bugs that are very hard to debug. 
public record ShipOrderCommand(
    @TargetAggregateIdentifier   // ✓ required!
    String orderId,
    String shippingAddress
) {}

Q 04
Event Sourcing — Events
Why must events be immutable and named in the past tense? Give one concrete example of each rule.
✦ Answer
Immutability — Events represent facts that already happened. Mutating a stored event would mean rewriting history, which destroys the entire audit trail and breaks aggregate state reconstruction (replay would produce wrong results).
→ Example: OrderCreatedEvent is a Java record — all fields are final by design.

Past tense — Events describe what occurred, not what you want. Using imperative names (CreateOrder) would confuse events with commands — a critical conceptual error in CQRS.
→ Example: Use OrderShippedEvent ✓, not ShipOrderEvent ✗. 
// ✓ Correct: past-tense, immutable record
public record OrderShippedEvent(
    String orderId,
    String shippingAddress
) {}

// ✗ Wrong: imperative name, mutable fields
public class ShipOrderEvent {
    public String orderId; // mutable — bad!
}

The Aggregate
Q 05
Axon — Aggregate
Look at this aggregate command handler. What is the critical mistake and why will it break Event Sourcing?
@CommandHandler
public OrderAggregate(CreateOrderCommand cmd) {
    // ← setting state directly — is this ok?
    this.orderId = cmd.orderId();
    this.status  = OrderStatus.CREATED;
    // no apply() call...
}
✦ Answer
Critical mistake: State is being set directly in the command handler instead of calling AggregateLifecycle.apply(event).

Why it breaks Event Sourcing:
  • No event is published → nothing is stored in the event store
  • The next time this aggregate is loaded, there are no events to replay → the aggregate starts completely empty — status = null
  • The entire audit trail is lost
State must only be set in @EventSourcingHandler methods, never in command handlers. 
// ✓ Correct
@CommandHandler
public OrderAggregate(CreateOrderCommand cmd) {
    apply(new OrderCreatedEvent(cmd.orderId(), ...)); // ← publish event
}

@EventSourcingHandler
public void on(OrderCreatedEvent e) {
    this.orderId = e.orderId();   // ← state set here ONLY
    this.status  = OrderStatus.CREATED;
}

Q 06
Axon — Aggregate
Why does an Axon aggregate need a protected no-arg constructor? When does Axon use it?
✦ Answer
Axon uses the no-arg constructor when loading an existing aggregate from the event store (i.e., when processing a command on an already-created order).

The process is:
  1. Axon calls the no-arg constructor to create an empty shell
  2. It fetches all stored events for that aggregate ID from the event store
  3. It replays them one by one through the @EventSourcingHandler methods
  4. After replay, the aggregate is fully hydrated and ready to handle the new command
Without the no-arg constructor, Java/Axon cannot instantiate the aggregate before replaying and throws an exception. 
@Aggregate
public class OrderAggregate {

    // Required! Axon calls this before event replay
    protected OrderAggregate() {}

    // This constructor is only for CREATE command
    @CommandHandler
    public OrderAggregate(CreateOrderCommand cmd) { ... }
}

Read Side
Q 07
CQRS — Projection
Explain the difference between @EventSourcingHandler and @EventHandler in Axon. Can they be in the same class?
✦ Answer
They look similar but serve very different purposes:

@EventSourcingHandler
→ Lives inside an @Aggregate class
→ Called to mutate aggregate state during command processing AND event replay
→ Part of the write side

@EventHandler
→ Lives in a plain Spring @Component (e.g., a Projection)
→ Called to update read model data (JPA entity, Redis, etc.)
→ Part of the read side, runs asynchronously via the EventBus

Can they be in the same class? No — @EventSourcingHandler only works inside @Aggregate. @EventHandler works in regular Spring beans. Mixing them would violate the CQRS separation. 
// WRITE SIDE — inside @Aggregate
@EventSourcingHandler
public void on(OrderCreatedEvent e) {
    this.status = OrderStatus.CREATED; // mutate aggregate state
}

// READ SIDE — inside @Component projection
@EventHandler
public void on(OrderCreatedEvent e) {
    repo.save(new OrderView(e.orderId(), "CREATED")); // update read DB
}

Q 08
Axon — Query
How do you dispatch a query in Axon and get back a single result? Write the controller code for fetching an order by ID.
✦ Answer
Use QueryGateway.query(), passing the query object and a ResponseTypes descriptor that tells Axon the expected return type. It returns a CompletableFuture — call .join() to block and get the result synchronously. 
// 1. Query object
public record FindOrderByIdQuery(String orderId) {}

// 2. Query handler (in projection)
@QueryHandler
public OrderView handle(FindOrderByIdQuery q) {
    return repo.findById(q.orderId()).orElseThrow();
}

// 3. Controller — dispatch the query
@GetMapping("/{id}")
public OrderView getOrder(@PathVariable String id) {
    return queryGateway.query(
        new FindOrderByIdQuery(id),
        ResponseTypes.instanceOf(OrderView.class)  // expected type
    ).join();  // block and return
}

Advanced — Saga & Snapshots
Q 09
Axon — Saga
In the guide's OrderProcessingSaga, what does associationProperty = "orderId" mean, and why is it essential?
@StartSaga
@SagaEventHandler(associationProperty = "orderId")
public void on(OrderCreatedEvent event) { ... }
✦ Answer
associationProperty = "orderId" tells Axon to link this saga instance to the value of the orderId field in the incoming event.

Why it's essential: There can be thousands of saga instances running concurrently (one per order). When a new event arrives (e.g., OrderConfirmedEvent for order #42), Axon needs to know which specific saga instance should handle it — it looks up sagas associated with orderId = "42".

Without associationProperty, Axon cannot correlate events to their saga instances, and the saga can never progress past the first event. 
// Axon internally does this lookup:
// "Find all saga instances where orderId = event.orderId()"
// → routes the event to the correct saga instance only

@SagaEventHandler(associationProperty = "orderId")
public void on(OrderConfirmedEvent e) {
    // This runs ONLY for the saga associated with e.orderId()
    commandGateway.send(new ShipOrderCommand(e.orderId(), ...));
}

Q 10
Event Sourcing — Snapshots
An order aggregate has accumulated 200 events. Explain step by step what happens when Axon loads it — both without and with a snapshot configured at every 50 events.
✦ Answer
Without snapshot (naive replay):
  1. Axon creates an empty aggregate (no-arg constructor)
  2. Fetches all 200 events from the event store
  3. Calls @EventSourcingHandler 200 times in sequence
  4. Aggregate is finally ready — but this is slow at scale
With snapshot (every 50 events):
  1. After event #50, Axon automatically saves a snapshot (full serialized aggregate state)
  2. Same at #100, #150, #200
  3. Next load: Axon finds the latest snapshot (at event #200 in this case)
  4. Deserializes the snapshot directly — no event replay needed at all
  5. Aggregate is ready instantly
Without new events after the snapshot, load time drops from O(N) to O(1). 
// Config: snapshot every 50 events
@Bean
public SnapshotTriggerDefinition orderSnapshotTrigger(Snapshotter s) {
    return new EventCountSnapshotTriggerDefinition(s, 50);
}

// Wire to aggregate
@Aggregate(snapshotTriggerDefinition = "orderSnapshotTrigger")
public class OrderAggregate { ... }

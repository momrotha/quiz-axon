02domain

Domain Model — What Are We Building?

Before writing any Axon code, understand the domain: an Order with a lifecycle from creation to shipping.

📦 The Order Domain

Our Order aggregate has a simple lifecycle:

CREATED → CONFIRMED → SHIPPED

Each state transition is triggered by a Command and recorded as an Event. The aggregate enforces business rules — for example, you cannot ship an order that hasn't been confirmed.

package com.example.order.domain;

public enum OrderStatus {

In DDD (Domain Driven Design), an Aggregate is a consistency boundary. The Order is our aggregate root — all business rules are enforced here, and it's the only way to change order state.

Step 2 of 12
03command

Commands — Express Intent

Commands are imperative messages that express what you want to happen. They are the entry point to the write side.

📨 What is a Command?

A command is a simple immutable data object (record or class with final fields). It is named in the imperative tense: CreateOrder, ConfirmOrder, ShipOrder.

The most important field is the @TargetAggregateIdentifier — this tells Axon which aggregate instance should receive the command.

CreateOrderCommand.javaJava 
package com.example.order.command;

import org.axonframework.modelling.command.TargetAggregateIdentifier;
import java.math.BigDecimal;

public record CreateOrderCommand(

    @TargetAggregateIdentifier   // ① tells Axon which aggregate to route to
    String orderId,

    String customerId,            // ② who placed the order
    BigDecimal totalAmount        // ③ order value
) {}


// ─────────────────────────────────────────────────

public record ConfirmOrderCommand(

    @TargetAggregateIdentifier
    String orderId               // just the ID to confirm
) {}


// ─────────────────────────────────────────────────

public record ShipOrderCommand(

    @TargetAggregateIdentifier
    String orderId,

    String shippingAddress        // where to ship
) {}
①
@TargetAggregateIdentifier — Required! Axon uses this field value to find (or create) the correct aggregate instance in the repository.
②
Commands carry all the data needed to perform the action. Think of them as a structured intent.
③
Using Java records makes commands naturally immutable — perfect for CQRS.
⚠️

Rule: Commands must be serializable (Axon serializes them when dispatching). Records with primitive/String fields work out of the box. Use Lombok @Value as an alternative.

Step 3 of 12
04events

Events — Facts That Happened

Events record what happened in the domain. They are past-tense, immutable, and stored forever in the event store.

📜 What is an Event?

Events are facts — they describe something that already occurred. Unlike commands, events cannot be rejected. Once published, they are persisted and broadcast.

Events are the source of truth in Event Sourcing. The aggregate state is rebuilt by replaying them in order.

OrderEvents.javaJava 
package com.example.order.event;

import java.math.BigDecimal;

/**
 * ① Past tense naming: OrderCREATED, OrderCONFIRMED, OrderSHIPPED
 * ② All fields final — events are IMMUTABLE
 * ③ Every event carries the aggregate ID for correlation
 */

public record OrderCreatedEvent(
    String orderId,
    String customerId,
    BigDecimal totalAmount
) {}


public record OrderConfirmedEvent(
    String orderId
) {}


public record OrderShippedEvent(
    String orderId,
    String shippingAddress
) {}
①
Past-tense naming — OrderCreated (not CreateOrder). This makes it clear: this is a fact, not a request.
②
Immutable — events must never change after creation. Records guarantee this in Java.
③
Always include orderId — projections and sagas need it to know which order this event belongs to.
💡

Event Sourcing key insight: There is no UPDATE or DELETE in your database. Instead, you store what happened. The current state of an Order is always computed by replaying all its events from the beginning.

Step 4 of 12
05aggregate

The Aggregate — Write Side Core

The aggregate is the heart of the write side. It enforces business rules and produces events in response to commands.

🏛️ What is an Aggregate in Axon?

An Axon @Aggregate is a DDD aggregate root that Axon manages. It receives commands, validates business rules, and publishes events using AggregateLifecycle.apply().

Its state is never saved directly to a database. Instead, Axon stores the events and rebuilds state by replaying them via @EventSourcingHandler methods.

OrderAggregate.javaJava 
package com.example.order.aggregate;

import org.axonframework.commandhandling.CommandHandler;
import org.axonframework.eventsourcing.EventSourcingHandler;
import org.axonframework.modelling.command.AggregateIdentifier;
import static org.axonframework.modelling.command.AggregateLifecycle.apply;
import org.axonframework.spring.stereotype.Aggregate;

@Aggregate                       // ① Mark as Axon-managed aggregate
public class OrderAggregate {

    @AggregateIdentifier         // ② The unique ID of this aggregate instance
    private String orderId;

    private OrderStatus status;   // ③ State — ONLY mutated in @EventSourcingHandler

    // ④ Required no-arg constructor for Axon to instantiate during replay
    protected OrderAggregate() {}

    // ════════════════════════════════
    // COMMAND HANDLERS
    // ════════════════════════════════

    @CommandHandler               // ⑤ Handles CreateOrderCommand → produces event
    public OrderAggregate(CreateOrderCommand cmd) {
        // Validate business rules here ↓
        if (cmd.totalAmount().signum() <= 0) {
            throw new IllegalArgumentException("Amount must be positive");
        }
        // Publish event — do NOT set state here directly!
        apply(new OrderCreatedEvent(
            cmd.orderId(), cmd.customerId(), cmd.totalAmount()
        ));
    }

    @CommandHandler
    public void handle(ConfirmOrderCommand cmd) {
        if (status != OrderStatus.CREATED) {         // ⑥ Enforce invariant
            throw new IllegalStateException("Only CREATED orders can be confirmed");
        }
        apply(new OrderConfirmedEvent(cmd.orderId()));
    }

    @CommandHandler
    public void handle(ShipOrderCommand cmd) {
        if (status != OrderStatus.CONFIRMED) {         // ⑥ Another invariant
            throw new IllegalStateException("Only CONFIRMED orders can be shipped");
        }
        apply(new OrderShippedEvent(cmd.orderId(), cmd.shippingAddress()));
    }
}
①
@Aggregate — Registers this class with Axon's repository. Axon handles loading, saving, and routing automatically.
②
@AggregateIdentifier — This field is used by the CommandBus to route commands to the exact aggregate instance.
③
State fields (status) are only ever set inside @EventSourcingHandler methods — never in command handlers.
④
The no-arg constructor is required by Axon to instantiate the aggregate before replaying events during hydration.
⑤
Constructor @CommandHandler — used for the creation command. Instance @CommandHandlers are for subsequent commands.
⑥
Business invariants — guard clauses ensure business rules are enforced before any event is emitted.

Let's trace exactly what happens when you dispatch a CreateOrderCommand through Axon.

🔄 The Command → Event Flow

When you call commandGateway.send(new CreateOrderCommand(...)), here is what Axon does internally — step by step:

① CommandGateway
.send(command)
→
Wraps command in a CommandMessage and dispatches to CommandBus
② CommandBus
routes command
→
Reads @TargetAggregateIdentifier, finds the matching @CommandHandler method
③ Repository

→
For existing aggregates: loads from event store and replays events
④ @CommandHandler
executes

Business rules validated, AggregateLifecycle.apply(event) called
⑤ Event persisted
+ published

Axon saves event to event store AND calls @EventSourcingHandler immediately
⑥ @EventHandler
on projection

💡

Golden rule: Command handlers only call apply(). They never set this.status = ... directly. State changes happen exclusively in @EventSourcingHandler methods. This is what makes event sourcing work.

Step 6 of 12
07ES handler

@EventSourcingHandler — Rebuilding State

These methods are the only place where aggregate state is mutated. They run during both command processing AND event replay.

🔁 The Event Sourcing Handler

When Axon stores an event (via apply()), it immediately calls the matching @EventSourcingHandler to update the aggregate's in-memory state. When Axon later needs to reload the aggregate, it replays all stored events through these same handlers — no database reads for state, just event replay.

OrderAggregate.java — continuedJava 
// Add these methods INSIDE OrderAggregate class
// (after the @CommandHandler methods)

// ════════════════════════════════
// EVENT SOURCING HANDLERS
// State mutation happens ONLY here
// ════════════════════════════════

@EventSourcingHandler            // ① Called when OrderCreatedEvent is applied
public void on(OrderCreatedEvent event) {
    this.orderId  = event.orderId();   // ② Set the aggregate ID
    this.status   = OrderStatus.CREATED; // ③ Set initial state
}

@EventSourcingHandler
public void on(OrderConfirmedEvent event) {
    this.status = OrderStatus.CONFIRMED; // ④ Transition state
}

@EventSourcingHandler
public void on(OrderShippedEvent event) {
    this.status = OrderStatus.SHIPPED;  // ④ Transition state
}
①
@EventSourcingHandler — matched by event type. Axon calls the right handler automatically based on the event class.
②
The first handler must set this.orderId — Axon needs it to correlate the aggregate identity during replay.
③④
Only state mutations here — no business logic, no validation, no side effects. Keep these methods simple and pure.
ℹ️

During event replay (e.g., loading an existing order), Axon calls the @EventSourcingHandler methods in sequence for every stored event. After replaying all 3 events above, the aggregate's status will be SHIPPED — the correct current state.

08projection

Projection — Building the Read Model

Projections listen to events and build a query-optimized view. This is the read side of CQRS.

📊 What is a Projection?

A projection is a Spring @Component that uses @EventHandler to listen to domain events and update a read model (a JPA entity, Redis hash, Elasticsearch document, etc.).

It has no business logic — it just transforms events into a format optimized for querying.

OrderView.javaJava — Read Model Entity 
@Entity
@Table(name = "order_view")
public class OrderView {

    @Id
    private String orderId;
    private String customerId;
    private BigDecimal totalAmount;
    private String status;
    private String shippingAddress;

    // constructors, getters, setters...
}
OrderProjection.javaJava 
@Component
@ProcessingGroup("order-projection")   // ① Logical group name for this processor
public class OrderProjection {

    private final OrderViewRepository repo;

    public OrderProjection(OrderViewRepository repo) {
        this.repo = repo;
    }

    @EventHandler                          // ② Reacts to OrderCreatedEvent
    public void on(OrderCreatedEvent e) {
        var view = new OrderView();
        view.setOrderId(e.orderId());
        view.setCustomerId(e.customerId());
        view.setTotalAmount(e.totalAmount());
        view.setStatus("CREATED");
        repo.save(view);                    // ③ Persist to read DB
    }

    @EventHandler
    public void on(OrderConfirmedEvent e) {
        repo.findById(e.orderId())
            .ifPresent(view -> {
                view.setStatus("CONFIRMED");
                repo.save(view);
            });
    }

    @EventHandler
    public void on(OrderShippedEvent e) {
        repo.findById(e.orderId())
            .ifPresent(view -> {
                view.setStatus("SHIPPED");
                view.setShippingAddress(e.shippingAddress());
                repo.save(view);
            });
    }
}
①
@ProcessingGroup — groups this handler into a named processor. You can configure threading and reset behavior per group.
②
@EventHandler — Axon delivers events here asynchronously. This is the read side reacting to write side changes.
③
The projection writes to a separate read database. Queries hit this table, never the event store. This is CQRS in action.

09query

Queries & Query Handlers

Queries are the read side of CQRS. They fetch data from the projection — never from the event store.

OrderQueries.java + handlersJava 
// ── Query objects ────────────────────────────────────
public record FindOrderByIdQuery(String orderId) {}
public record FindAllOrdersQuery() {}


// ── Add these to OrderProjection.java ────────────────

@QueryHandler                      // ① Handles FindOrderByIdQuery
public OrderView handle(FindOrderByIdQuery query) {
    return repo.findById(query.orderId())
               .orElseThrow(() ->
                   new IllegalArgumentException("Order not found"));
}

@QueryHandler                      // ② Handles FindAllOrdersQuery
public List<OrderView> handle(FindAllOrdersQuery query) {
    return repo.findAll();
}
ℹ️

Queries are dispatched via QueryGateway, not CommandGateway. The QueryGateway routes the query object to the matching @QueryHandler method by type — just like commands.

10saga

Saga — Long-Running Process Orchestration

A Saga coordinates a multi-step business process that spans multiple aggregates or services over time.

🎭 What is a Saga?

A Saga is a stateful event listener that reacts to events and dispatches commands to drive a process forward. In our example: when an order is created, the saga automatically sends a payment request.

OrderProcessingSaga.javaJava 
@Saga                                    // ① Mark as Axon Saga
public class OrderProcessingSaga {

    @Autowired
    private transient CommandGateway commandGateway; // ② transient = not stored

    private String orderId;

    @StartSaga                             // ③ Creates new saga instance
    @SagaEventHandler(associationProperty = "orderId")
    public void on(OrderCreatedEvent event) {
        this.orderId = event.orderId();
        // Saga sends a command to another aggregate/service
        commandGateway.send(new ConfirmOrderCommand(event.orderId()));
        // In real life: send RequestPaymentCommand to PaymentAggregate
    }

    @SagaEventHandler(associationProperty = "orderId")
    public void on(OrderConfirmedEvent event) {
        // React to confirmation — maybe trigger shipping prep
        commandGateway.send(
            new ShipOrderCommand(event.orderId(), "Default Warehouse")
        );
    }

    @EndSaga                               // ④ Marks saga as complete → cleaned up
    @SagaEventHandler(associationProperty = "orderId")
    public void on(OrderShippedEvent event) {
        // Process complete! Saga instance is removed from saga store.
    }
}
①
@Saga — tells Axon this is a saga. Axon persists saga state in the saga store between events.
②
transient — Spring-injected dependencies must be transient so they are not serialized with the saga state.
③
@StartSaga — creates a new saga instance. associationProperty links this saga to a specific orderId.
④
@EndSaga — marks the saga complete. Axon removes it from the saga store, freeing resources.

11snapshots

Snapshots — Performance Optimization

As aggregates accumulate hundreds of events, replaying all of them on each load becomes slow. Snapshots solve this.

📸 How Snapshots Work

A snapshot captures the full aggregate state at a given event sequence number. On next load, Axon loads the snapshot + only the events after the snapshot — skipping potentially thousands of old events.

AxonConfig.javaJava — Snapshot Configuration 
@Configuration
public class AxonConfig {

    /**
     * Take a snapshot every 50 events applied to an aggregate.
     * After 50 events → snapshot stored → next load uses snapshot + new events only.
     */
    @Bean
    public SnapshotTriggerDefinition orderSnapshotTrigger(
            Snapshotter snapshotter) {

        return new EventCountSnapshotTriggerDefinition(
                snapshotter,
                50              // ← snapshot after every 50 events
        );
    }
}
OrderAggregate.java — wire snapshot triggerJava 
// Reference the snapshot trigger bean by name in @Aggregate
@Aggregate(snapshotTriggerDefinition = "orderSnapshotTrigger")
public class OrderAggregate {
    // ... rest of the aggregate
}

12REST API

REST Controller — Wiring It All Together

The final piece: expose HTTP endpoints that dispatch commands and queries through Axon gateways.

OrderController.javaJava 
@RestController
@RequestMapping("/orders")
public class OrderController {

    private final CommandGateway commandGateway;  // ① Write side
    private final QueryGateway   queryGateway;    // ② Read side

    public OrderController(CommandGateway cg, QueryGateway qg) {
        this.commandGateway = cg;
        this.queryGateway   = qg;
    }

    // ── WRITE ENDPOINTS ────────────────────────────────

    @PostMapping
    public String createOrder(@RequestBody CreateOrderRequest req) {
        String id = UUID.randomUUID().toString();
        commandGateway.sendAndWait(                  // ③ Dispatch command
            new CreateOrderCommand(id, req.customerId(), req.totalAmount())
        );
        return id;
    }

    @PutMapping("/{id}/confirm")
    public void confirmOrder(@PathVariable String id) {
        commandGateway.sendAndWait(new ConfirmOrderCommand(id));
    }

    @PutMapping("/{id}/ship")
    public void shipOrder(@PathVariable String id,
                           @RequestParam String address) {
        commandGateway.sendAndWait(new ShipOrderCommand(id, address));
    }

    // ── READ ENDPOINTS ─────────────────────────────────

    @GetMapping("/{id}")
    public OrderView getOrder(@PathVariable String id) {
        return queryGateway.query(                    // ④ Dispatch query
            new FindOrderByIdQuery(id),
            ResponseTypes.instanceOf(OrderView.class)
        ).join();
    }

    @GetMapping
    public List<OrderView> getAllOrders() {
        return queryGateway.query(
            new FindAllOrdersQuery(),
            ResponseTypes.multipleInstancesOf(OrderView.class)
        ).join();
    }
}
①
CommandGateway — the entry point to the write side. sendAndWait() dispatches synchronously and throws on failure.
②
QueryGateway — the entry point to the read side. It routes the query to the matching @QueryHandler.
③
sendAndWait() blocks until the command is fully processed. Use send() for fire-and-forget async behavior.
④
ResponseTypes tells Axon what return type to expect from the @QueryHandler.
